FLAGSHIP CASE STUDY

Flagship: Retail Demand and Inventory Pipeline

End-to-End Delivery
Public Dataset ELT Pipeline Star Schema Power BI Output View GitHub Artifacts

Problem Statement

Retail analysts needed dependable visibility into demand, stock movement, and replenishment risk, but source data arrived in inconsistent files and required manual consolidation before reporting.

Approach

Implemented an ELT pipeline with staged ingestion, schema validation, SQL-first transformations, and a dimensional model designed for both operational and executive analytics.

Tools Used

Public retail datasets, Azure Data Factory patterns, SQL transformation models, Python validation scripts, and Power BI semantic reporting with incremental refresh configuration.

Outcome / Impact

Delivered a repeatable pipeline and governed model that replaced manual report assembly with a traceable data product. Validation checks made data issues visible before dashboard publication.

Data Source Context

Public retail demand and order datasets used to simulate production-like ingestion, schema drift handling, and incremental loads.

Modeling Decision

Star schema with fact_orders and dimensions for product, customer, store, and calendar to support reusable analytics across teams.

Validation Strategy

Row-count reconciliation, null threshold checks, duplicate-key detection, and publish gates for failed quality rules.

[ Architecture Flow ]

1) Raw Data Intake → Landing Zone (versioned files)
2) Bronze Layer → immutable append-only records with ingestion timestamp
3) Silver Layer → deduplication, conformed keys, business rule transforms
4) Gold Layer → fact and dimension models for demand and inventory analytics
5) Semantic Layer → Power BI model with governed KPI definitions
-- Incremental transform using watermark and merge DECLARE @last_success_ts DATETIME2 = ( SELECT MAX(successful_load_ts) FROM etl_audit WHERE pipeline_name = 'retail_orders_silver' ); WITH delta_incoming AS ( SELECT * FROM bronze_orders WHERE ingestion_ts > @last_success_ts ) MERGE INTO silver_orders AS tgt USING delta_incoming AS src ON tgt.order_id = src.order_id WHEN MATCHED THEN UPDATE SET tgt.amount = src.amount, tgt.updated_at = GETDATE() WHEN NOT MATCHED THEN INSERT (order_id, customer_id, product_id, amount, order_date, updated_at) VALUES (src.order_id, src.customer_id, src.product_id, src.amount, src.order_date, GETDATE());
# Python quality gate executed before Gold publish def run_quality_checks(df): required_cols = {"order_id", "customer_id", "order_date", "amount"} missing = required_cols - set(df.columns) if missing: raise ValueError(f"Missing required columns: {sorted(missing)}") duplicate_orders = df.duplicated(["order_id"]).sum() if duplicate_orders > 0: raise ValueError(f"Duplicate order_id records: {duplicate_orders}") null_amount = df["amount"].isna().sum() if null_amount > 0: raise ValueError(f"Null amount records: {null_amount}") revenue_outliers = (df["amount"] < 0).sum() if revenue_outliers > 0: raise ValueError(f"Negative amount rows: {revenue_outliers}")

// Case Study Deep Dive

From Raw Files to Trusted Decisions
End-to-End Engineering Reasoning

This section shows how I think through pipeline design choices, data modeling, and performance tradeoffs on a realistic workload.

Before

Manual spreadsheet consolidation, inconsistent business rules between teams, and no reliable way to trace errors back to source records.

After

A layered data product with quality gates, conformed dimensions, and governed KPI logic that can be re-used across reporting consumers.

Pipeline Flow and Engineering Decisions

  1. 1. Ingestion: landed all source files with immutable timestamps so reruns do not overwrite source history.
  2. 2. Standardization: normalized keys and date formats, then quarantined invalid records for review instead of dropping them silently.
  3. 3. Transformation: applied SQL business rules in Silver models to keep transformations reviewable and version controlled.
  4. 4. Modeling: built star schema entities to separate operational transaction detail from analytics-ready aggregates.
  5. 5. Serving: exposed curated Gold views to Power BI with clear semantic definitions for shared KPIs.

Performance and Reliability Focus

  • Partition-aware loading strategy to avoid full refresh cycles.
  • Incremental merge logic controlled by pipeline watermark state.
  • Publish blocking when data quality thresholds fail.
  • Audit table for every load run to support incident response.

Business Value Delivery

  • Reduced dependence on manual reconciliation steps.
  • Improved consistency of demand and inventory reporting.
  • Faster access to curated data for analyst and operations teams.
  • Stronger trust in dashboard outputs through traceable lineage.
